## Intent

Fix the demo feedback system's identity, deduplication, session-duration, and
serverless rate-limit gaps. Produce a reusable AI prompt for replicating the
entire feature in another codebase.

## Story And Acceptance Criteria

As the ChurchCore platform team, we need feedback records to carry trustworthy
context and resist accidental or abusive duplication so staff can triage demo
reports accurately across distributed deployments.

Acceptance criteria:

- Authenticated identity is derived by the server; anonymous reports stay
  anonymous and browser-provided identity is ignored.
- Fingerprints are generated by the server.
- Equivalent errors and manual notes deduplicate, while different manual notes
  remain separate.
- Session duration is validated, stored, and visible to platform staff.
- A control-plane Postgres function atomically enforces 20 submissions per
  session per 60 seconds across app instances.
- Duplicate reports increment hit count and reopen completed triage.
- The legacy submission function is converted to invoker rights and restricted
  to `service_role` for rolling-deploy compatibility.
- Demo-mode gating and existing staff triage behavior remain intact.
- A self-contained cross-codebase implementation prompt is documented.

## Technical Brief

- The browser sends observation context only.
- `POST /api/demo/feedback` validates bounded fields, resolves the current
  session, computes normalized SHA-256 fingerprints, hashes the limiter key, and
  invokes one control-plane service-role RPC.
- `submit_demo_feedback` serializes each limiter key with a transaction advisory
  lock, updates the fixed window, and upserts accepted feedback in the same
  transaction. It uses invoker rights and is executable only by `service_role`.
- Feedback and limiter state remain in the control-plane database. No tenant
  operational table or tenant service-role client is used.
- Stored identity is limited to profile email and role ID. No IP address, token,
  stack trace, or raw authentication payload is retained.

## Residual Risk

- Browser-generated session IDs can be rotated. The limiter is reliable across
  serverless instances but is not full bot protection.
- The migration is compatible with the currently deployed API and should still
  be applied to the control-plane project before activating the new API code.
- Live database concurrency behavior is encoded in Postgres but has not yet been
  exercised against a connected control-plane project in this run.
